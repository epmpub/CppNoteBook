# Folly 到底解决了 C++ 标准库的哪些痛点？



下面是**最核心、最常用、工程里真能用上**的 8 大类痛点，每一条都是 Meta 线上万亿级流量验证过的。

------

# 一、哈希表：std 太慢、内存碎片、缓存不友好

**std 问题：**

- `std::unordered_map` 链式哈希，**缓存不友好**
- 迭代慢、查询慢、内存占用大
- 没有并发优化

**Folly 解决方案：**

- **F14 哈希表家族**（F14NodeMap / F14ValueMap / F14FastMap）
- **速度比 std 快 2~10 倍**
- 内存更小、缓存命中率极高
- 支持迭代器稳定性、无指针开销、SIMD 搜索

**一句话：F14 就是 C++ 哈希表的标准答案。**

------

# 二、线程池 & 异步执行：std 完全没有好用的池

**std 问题：**

- 只有 `std::thread` / `std::async`，**没有线程池**
- 无法控制线程数、无法复用、无法优先级调度
- 无法做 IO 密集型调度

**Folly 解决方案：**

- `CPUThreadPoolExecutor`
- `IOThreadPoolExecutor`
- 优先级调度、动态伸缩、队列策略、保活机制
- 协程 + Future 深度集成

**一句话：std 没有线程池生态，Folly 直接补全工业级能力。**

------

# 三、异步 & 协程：std 异步太弱，没有真正的协程

**std 问题：**

- C++20 协程太底层，**无法直接用**
- `std::future` 无法链式、无法回调、无法组合

**Folly 解决方案：**

- `folly::Future/Promise` 链式异步编程
- 完备的协程 runtime
- 高并发网络框架
- 异步 I/O、定时器、事件循环

**一句话：Folly 让 C++ 真正能写高并发服务。**

------

# 四、字符串处理：std 又难用又慢

**std 问题：**

- 没有 split、join、trim、startswith、endswith
- 格式化繁琐（C++20 才勉强补上）
- 性能一般

**Folly 解决方案：**

- `folly::split()`
- `folly::join()`
- `folly::stringPrintf()`
- `folly::to<string>()`
- 安全、快速、一行搞定

**一句话：字符串工具直接补齐现代语言级别的体验。**

------

# 五、内存 & 零拷贝：std 完全没有

**std 问题：**

- 没有专门为网络设计的缓冲区
- 大量拷贝，性能差

**Folly 解决方案：**

- `folly::IOBuf` —— **零拷贝网络 buffer**
- 链式内存、引用计数、不复制数据
- 行业标准，所有 C++ 高性能网络都用它

------

# 六、并发原语：std 不够强

**std 问题：**

- 原子操作、锁、线程局部存储都不够高效
- 高并发下性能一般

**Folly 解决方案：**

- `folly::MicroSpinLock`
- `folly::ThreadLocal`
- `folly::AtomicHashMap`
- 高并发场景性能远超 std

------

# 七、时间、时钟、超时：std 难用

**std 问题：**

- 接口复杂、容易错、不直观

**Folly 解决方案：**

- `folly::chrono`
- `folly::stopwatch`
- 简单、安全、不易出错

------

# 八、工具类：std 缺失太多日常功能

Folly 补全了一大堆现代语言必备工具：

- 动态数组
- 位操作
- base64 / 哈希 / 校验
- 配置解析
- 随机数
- 异常处理
- 断言 & 日志工具

------

# 终极总结（最精炼版）

## Folly 解决了 C++ 标准库 4 大核心缺陷：

1. **性能差**（哈希表、内存、并发、字符串）
2. **功能缺**（线程池、协程、零拷贝、IO 调度）
3. **接口难用**（字符串、时间、并发工具）
4. **不适合高并发服务**（网络、异步、大规模分布式）

------

## 一句话记住 Folly：

**Folly = 更快的标准库 + 缺失的高并发库 + 现代 C++ 工程工具箱**
