C++23 字符串与文本处理

## 1. `contains` 成员函数

`std::basic_string`（即 `std::string`）和 `std::basic_string_view`（即 `std::string_view`）新增了 `contains` 成员函数，用于判断字符串中是否包含某个子串/字符，语义上等价于 `find(...) != npos`，但更直观易读。

```cpp
std::string s = "hello world";
if (s.contains("world")) { /* ... */ }      // 子串
if (s.contains('w')) { /* ... */ }          // 单个字符
if (s.contains(std::string_view{"wor"})) {} // string_view
```

C++23 之前只能写 `s.find("world") != std::string::npos`，容易写错（比如忘了跟 `npos` 比较），`contains` 消除了这个常见 bug 来源，也和 `std::set`/`std::map` 等容器在 C++20 中新增的 `contains` 保持一致的命名习惯。

------

## 2. 禁止从 `nullptr` 构造

C++23 之前，下面这段代码能**编译通过**，但在运行期是未定义行为（因为 `nullptr` 会先隐式转换成 `const char*`，然后传给以 `const char*` 为参数、内部调用 `strlen` 的构造函数）：

```cpp
std::string s = nullptr;       // C++23 之前：编译通过，运行期 UB（崩溃）
std::string_view sv = nullptr; // 同样问题
```

C++23 显式地删除（`= delete`）了接受 `nullptr_t` 的构造函数重载，把这个错误从**运行期崩溃**提前到**编译错误**：

```cpp
std::string s = nullptr;       // C++23 起：编译错误
std::string_view sv = nullptr; // C++23 起：编译错误
```

这是一个纯粹的**安全性修正**，防止一类常见的空指针误用。

------

## 3. `basic_string_view` 的显式 range 构造函数

C++20 给 `std::string_view` 增加了从任意"范围"（满足 `contiguous_range` 且元素类型兼容的对象，比如自定义的连续字符容器）构造的能力。但这个构造函数最初是**隐式**的，可能导致意外的隐式转换（例如把一个本不该被当作 `string_view` 的容器悄悄转换过去，产生生命周期或语义上的隐患）。

C++23 把这个 range 构造函数改为 `explicit`：

```cpp
std::vector<char> v = {'a', 'b', 'c'};

std::string_view sv1 = v;             // C++23 起：编译错误（不再允许隐式转换）
std::string_view sv2{v};              // OK，显式构造
std::string_view sv3(v.begin(), v.end()); // 也可以，走迭代器对构造
```

这样可以避免代码里出现"看起来是普通赋值，实际发生了潜在危险的隐式转换"的情况，强制程序员显式表达意图。

------

## 4. `resize_and_overwrite`

**动机**：`std::string::resize(n)` 在扩容时，标准要求新增的字符必须先被**值初始化**（通常是填充 `'\0'`），然后你才能通过 `data()`/`operator[]` 去覆写这些字符。这意味着如果你打算立刻整体覆盖这块新内存（比如用 `snprintf`、自定义写入逻辑填充数据），标准的 `resize` 会强制你**先做一次多余的初始化**，浪费性能。

`resize_and_overwrite` 允许你传入一个"填充器"可调用对象，跳过这次多余的初始化：

```cpp
std::string s;
s.resize_and_overwrite(64, [](char* buf, std::size_t n) {
    // buf 指向未初始化（或者说：内容未指定）的 n 个字符
    std::size_t written = std::snprintf(buf, n, "value=%d", 42);
    return written; // 返回实际写入的长度，字符串会被截断/resize 到这个长度
});
```

签名大致为：

```cpp
template<class Op>
void resize_and_overwrite(size_type n, Op op);
// op 被调用为: op(data(), n) -> 返回值必须可转换为 size_type，表示最终的字符串长度
```

**典型应用**：性能敏感场景下，用 C 风格 API（如 `snprintf`）、自定义序列化逻辑等直接写入字符串底层 buffer，避免"先清零/初始化，再覆盖"的双重开销。这类似于很多高性能库（如 folly 的 `resizeWithoutInitialization`）里早已存在的自定义扩展，现在被标准化了。

------

## 5. `substr` 的右值引用重载

`std::string::substr(pos, len)` 传统上总是返回一个**新的字符串对象**（拷贝），即使原字符串是一个即将被丢弃的临时对象（右值）：

```cpp
std::string make_string();
std::string s = make_string().substr(0, 5); // C++23 之前：仍然发生一次拷贝
```

C++23 新增了一个**右值限定（rvalue-qualified）\**的 `substr` 重载，当在右值 `std::string` 上调用 `substr` 时，可以\**复用原字符串的内存**（把原字符串"移动并裁剪"），而不是重新分配、拷贝：

```cpp
// 概念上等价于新增了这样一个重载：
std::string substr(size_type pos = 0, size_type count = npos) &&;
std::string s = make_string().substr(0, 5); // C++23 起：可能直接复用/移动原 buffer，避免额外拷贝
```

这纯粹是一个**性能优化**：编译器/标准库会根据表达式的值类别（左值 vs 右值）自动选择合适的重载，用户代码写法不变，但在链式调用临时字符串时能省掉一次分配和拷贝。

注意：这只是"可能"更高效——具体是否真的避免了内存分配取决于标准库实现细节（比如是否直接原地 `erase` 尾部/头部并 `shrink`），但标准至少给了实现这样做的许可和接口基础。

------

## 小结

这一组改动都是围绕**字符串使用中的安全性和性能**展开的小而实用的改进：

- `contains`：提升可读性、减少 `find`/`npos` 误用；
- 禁止 `nullptr` 构造：把运行期 UB 变成编译期错误；
- `string_view` range 构造改为 `explicit`：防止意外的隐式转换带来生命周期风险；
- `resize_and_overwrite`：避免不必要的初始化开销，服务于高性能写入场景；
- `substr` 右值重载：链式调用临时字符串时减少不必要的拷贝。
