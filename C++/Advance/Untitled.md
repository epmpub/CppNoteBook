#### constexpr

------

**性能：不只是"算得快"**

运行时更快是最直观的，但更深层的收益是二进制体积的变化。编译期算好的常量直接嵌入指令流，不再占用 `.data` 段，也不需要在启动时构建查找表或哈希表。对于嵌入式、游戏引擎这类对 instruction cache 敏感的场景，减少运行时分支和间接跳转带来的收益有时比"跑得快"更重要。

------

**正确性：错误从运行时移到编译时**

这是现代 C++ 最看重的一点。`constexpr` 函数被用在常量表达式上下文（如 `static_assert`、模板参数、数组大小）时，编译器必须在编译期对它求值——求值失败就是编译错误，而不是运行时崩溃。

```cpp
constexpr int divide(int a, int b) {
    if (b == 0) throw std::invalid_argument("div by zero");
    return a / b;
}

static_assert(divide(10, 2) == 5);  // OK
static_assert(divide(10, 0) == 5);  // 编译错误，不是运行时异常
```

这意味着测试用例本身可以嵌入类型系统。你不需要等 CI 跑完才发现某个配置计算错了，编译一次就知道。

------

**安全：消除一整类运行时不确定性**

两个常被忽略的安全维度：

第一是静态存储期对象的初始化顺序问题（Static Initialization Order Fiasco）。全局对象的初始化顺序跨编译单元是未定义的，历史上这是 C++ 程序里很难排查的 bug 来源。`constexpr` 全局变量保证在编译期完成初始化，彻底绕开这个问题。

第二是消除 TOCTOU（Time-of-Check Time-of-Use）竞争窗口。运行时读取配置、计算哈希盐、生成查找表，都存在被竞争或被替换的窗口。编译期固化的值不存在这个窗口。

------

**工程：约束即文档，意图即测试**

`constexpr` 有一种非常有价值的副作用——它迫使你写出"纯"的计算逻辑。一个真正的 `constexpr` 函数不能有副作用、不能依赖全局状态、不能做 I/O，这些限制恰好是函数式编程推崇的属性，也让函数更容易测试和推理。

更妙的是，`static_assert` 把断言从测试框架拉到了语言本身里。以前你需要写单元测试验证某个编译期常量的值，现在直接：

```cpp
static_assert(std::numeric_limits<int>::max() == 2147483647);
static_assert(sizeof(MyProtocolHeader) == 16, "ABI break!");
```

这些"测试"永远不会被人删掉、跳过或忘记运行——它们是编译的一部分。

------

**一个更大的图景**

C++ 这条路线背后有一个统一的哲学："你不为不使用的东西付代价，你使用的东西能做到最优"（Bjarne Stroustrup 的零开销原则）。把计算推向编译期是这个原则的自然延伸：运行时只做运行时才能做的事，其余的都在构建时解决掉。

C++20/23/26 不断扩展 `constexpr` 边界（`std::string`、`std::vector`、反射、`consteval` 函数），本质上是在把"运行时开销"这条线越画越靠右，让程序员有更大的空间在不牺牲抽象能力的前提下实现零开销。
