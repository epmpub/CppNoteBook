# C++23 元编程相关工具

## 1. Comparison Concepts 支持 Move-only 类型

C++20 引入了比较相关的 concepts：`std::equality_comparable`、`std::totally_ordered`、`std::three_way_comparable` 等（以及它们的 `_with` 版本）。但 C++20 的定义里隐含要求被比较的对象**可以被拷贝**（因为内部实现用到了会产生临时对象拷贝的写法），导致像 `std::unique_ptr` 这样的 move-only 类型即使定义了合法的 `==`、`<=>`，也无法满足这些 concepts。

C++23（通过 **P2404**）修正了这些 concept 的定义方式，让它们不再要求可拷贝，只要求表达式合法：

```cpp
static_assert(std::equality_comparable<std::unique_ptr<int>>); // C++23 起为 true
static_assert(std::totally_ordered<std::unique_ptr<int>>);     // C++23 起为 true
```

这纯粹是**concept 定义的修正**，不需要用户改代码，`unique_ptr` 本身的比较运算符没变，是 concept 变"宽容"了。

## 2. 新增类型特征（Type Traits）

### `std::is_scoped_enum`

判断一个类型是否是 **scoped enum**（即 `enum class` / `enum struct`），区别于传统的 unscoped `enum`。

```cpp
enum class Color { Red, Green };
enum Old { A, B };

static_assert(std::is_scoped_enum_v<Color>);   // true
static_assert(!std::is_scoped_enum_v<Old>);    // false，是普通枚举
static_assert(!std::is_scoped_enum_v<int>);    // false，根本不是枚举
```

用途：比 `std::is_enum` 更精确，常用于模板元编程中区分两种枚举以决定是否允许隐式转换到底层整数类型等场景。

### `std::is_implicit_lifetime`

判断一个类型是否属于 **"隐式生命周期类型"（implicit-lifetime type）**——即那些**不需要显式调用构造函数**、可以通过如 `malloc`/`memcpy`/`std::start_lifetime_as` 等底层操作"隐式开始生命周期"的类型。这类类型典型的包括：标量类型、聚合类型（无用户定义构造函数的 struct/array）、trivially copyable 的类等。

```cpp
struct Aggregate { int x; double y; };
struct HasCtor { HasCtor() {} };

static_assert(std::is_implicit_lifetime_v<Aggregate>); // true
static_assert(std::is_implicit_lifetime_v<int>);       // true
static_assert(!std::is_implicit_lifetime_v<HasCtor>);  // false，有用户定义构造函数
```

这个 trait 是配合 C++23 新增的 `std::start_lifetime_as`（用于在**不运行构造函数**的前提下，将一段原始内存"重新解释"为一个对象，比如从 `mmap`/驱动返回的 buffer）而引入的，属于底层内存/对象模型相关的元编程工具。

### `std::reference_constructs_from_temporary` / `std::reference_converts_from_temporary`

这两个 trait 用于**编译期检测引用是否会绑定到一个临时对象上**——即检测潜在的"悬垂引用"（dangling reference）风险，这是 C++ 里一个经典的陷阱：

```cpp
const std::string& s = std::string("hello") + "world"; // 悬垂引用！临时对象生命周期结束后 s 变为悬垂
```

- `std::reference_constructs_from_temporary<T, U>`：检测用 `U` 类型的值**直接构造** `T` 类型引用时，是否会产生临时对象绑定（对应直接初始化语义，例如 `T ref(std::declval<U>())`）。
- `std::reference_converts_from_temporary<T, U>`：检测**隐式转换**场景下（对应复制初始化语义，例如 `T ref = std::declval<U>()`）是否会产生临时对象绑定。

两者语义相近，区别在于构造语法（`()`/`{}` 直接初始化）与转换语法（`=` 隐式转换）在重载决议上可能选到不同的构造函数/转换函数，因此需要分开定义。

```cpp
// 简化示例：int 转 double 引用会产生临时对象
static_assert(std::reference_constructs_from_temporary_v<const double&, int>); // true

// 同类型引用绑定同类型左值，不产生临时对象
static_assert(!std::reference_constructs_from_temporary_v<const int&, int&>); // false
```

**典型应用**：编译器/静态分析工具、以及库作者可以用这些 trait 在模板代码中加 `static_assert` 或 SFINAE，提前在编译期拒绝会产生悬垂引用的调用，而不是留下运行期未定义行为。事实上，GCC/Clang 在 C++23 后也用类似机制加强了对悬垂引用的编译期警告（如 `-Wdangling-reference`）。

## 小结

这几项都属于"让模板/元编程作者能在编译期**更精确地探查类型属性**"这一大方向：

- comparison concepts 的修正解决了 **move-only 类型被过度排除** 的问题；
- `is_scoped_enum` 和 `is_implicit_lifetime` 提供了更细粒度的类型分类，服务于泛型代码和底层内存操作；
- `reference_constructs/converts_from_temporary` 直接针对 C++ 中臭名昭著的**悬垂引用**问题，把原本只能运行期出错的隐患变成可以编译期检测的信息。

```c

#include <memory>
#include <cmath>
#include <charconv>
#include <type_traits>

constexpr int f() {
	auto p = std::make_unique<int>(42);
    return *p;
}

constexpr bool test() {
    char buf[10]{};
    auto res = std::to_chars(buf, buf + 10, 123);
    return std::string_view(buf, res.ptr) == "123";
}

enum Old { A, B };


int main() {
    constexpr std::bitset<8> b{ 0b1010'1010 };
    static_assert(b.count() == 4);
	static_assert(f() == 42);

    /*constexpr*/ double r = std::sqrt(4); // C++23 起合法,MSVC 不支持；

    static_assert(test()); // C++23 起合法（仅整数重载）

    static_assert(std::equality_comparable<std::unique_ptr<int>>); // C++23 起为 true
    static_assert(std::totally_ordered<std::unique_ptr<int>>);     // C++23 起为 true

    static_assert(std::is_scoped_enum_v<Color>);   // true
    static_assert(!std::is_scoped_enum_v<Old>);    // false，是普通枚举
    static_assert(!std::is_scoped_enum_v<int>);    // false，根本不是枚举

    struct Aggregate { int x; double y; };
    struct HasCtor { HasCtor() {} };

    //static_assert(std::is_implicit_lifetime_v<Aggregate>); // true

}
```
