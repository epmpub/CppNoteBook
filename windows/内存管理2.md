## Memory Fundamentals 记忆基础

You would expect that the documentation for, say, [VirtualAlloc](https://learn.microsoft.com/en-us/windows/win32/api/memoryapi/nf-memoryapi-virtualalloc) would get a little bit more specific than “If the function fails, the return value is NULL.” Let’s start by establishing the minimum knowledge necessary to be able to answer “in what situation do allocations fail?”
你会期待比如 [VirtualAlloc](https://learn.microsoft.com/en-us/windows/win32/api/memoryapi/nf-memoryapi-virtualalloc) 的文档会比“如果函数失败，返回值为 NULL”更具体一些。让我们先建立出回答“在什么情况下配置会失败？”这一问题所需的最低知识。

Here is a simplified model of memory management:
这里有一个简化的内存管理模型：

- Programs live in a virtual address space that does not necessarily represent physical reality. We call this “virtual memory.” Virtual memory is assigned and managed by the operating system (OS).
  程序存在于一个虚拟地址空间中，这个空间不一定代表物理现实。我们称之为“虚拟记忆”。虚拟内存由操作系统（OS）分配和管理。
- Virtual address space is divided into “pages” of a certain size (frequently 4 kilobytes, still). Pages have access-rights: can memory in this page be read? can it be written to? can it contain executable code?
  虚拟地址空间被划分为一定大小的“页面”（通常仍为4千字节）。页面具有访问权：该页面中的内存可以被读取吗？能写信给我吗？它能包含可执行代码吗？
- The OS ensures (along with some hardware) that when your program actually accesses a valid virtual memory address, there is some physical representation of the bytes you are accessing. This is done by “mapping” pages to a physical representation.
  操作系统确保（连同一些硬件）当你的程序实际访问一个有效的虚拟内存地址时，会有你所访问字节的某种物理表示。这是通过“映射”页面到物理表示来实现的。

Note, the term “physical representation” is not a standard term. I am just using it here. It’s common to say that a page has some physical “backing”, though in general terminology is loose and inconsistent in this entire space (as evidenced by the MSDN page [Memory Performance Information](https://learn.microsoft.com/en-us/previous-versions/windows/desktop/legacy/aa965225(v=vs.85))).
注意，“物理表征”一词并非标准术语。我只是在这里使用它。通常会说页面有一定的物理“背后”，但总体来说，这个空间的术语很宽松且不一致（正如 MSDN 页面“[ 内存性能信息 ](https://learn.microsoft.com/en-us/previous-versions/windows/desktop/legacy/aa965225(v=vs.85))”所示）。

That’s about what we need, and you probably knew as much already. Let me outline some “features” of such a memory systems:
这正是我们需要的，你大概早就知道了。让我来概述一下这些记忆系统的一些“特点”：

- It is very often convenient to manually divide up your address space by allocating virtual memory. You want a terabyte of memory that is guaranteed to have contiguous virtual addresses? Go for it. You can totally allocate more virtual memory than your machine has physical RAM, and even more than you may have disk space. Many systems allow you to allocate so much virtual memory that it is virtually guaranteed (badum-ts) that you do not have a physical representation for all of it.
  通过分配虚拟内存手动分割地址空间通常非常方便。你想要一个 TB 内存，并且保证拥有连续的虚拟地址？去吧。你完全可以分配比机器物理内存更多的虚拟内存，甚至超过你可能拥有的磁盘空间。许多系统允许你分配大量虚拟内存，几乎可以肯定（badum-ts）你没有所有虚拟内存的物理表示。
- The OS does not actually need to map a virtual memory page to some physical representation before you touch it. Maybe, like many programs, you *never* touch it? What a waste it would be to map that! Pages are hence mapped “on demand.”
  操作系统实际上不需要在你触碰虚拟内存页面之前，先将虚拟内存页面映射到某种物理表示。也许，像许多程序一样， *你根本不*碰它？绘制地图真是太浪费了！因此，页面被按需映射。
- The physical representation for a byte does not have to be RAM. It could be anything sufficiently “byte-shaped.”
  字节的物理表示不必是 RAM。它可以是任何足够“字节形状”的东西。
- Two different pages in virtual address space could be mapped to the same physical representation.
  虚拟地址空间中的两个不同页面可以映射到相同的物理表示。

Let’s talk about physical representations for a moment. Generally speaking, a page can be mapped to different things: It could be RAM. It could be different hardware with its own memory that you want conveniently addressable. It could be a file on disk. For general purpose allocations, there are two things that count: There is your RAM, and there is your page file (or swap space on Linux). Ideally, all the data you ever need would be in RAM, but in all likelihood your system will sometimes need to provide more bytes than you have RAM, so it will write parts of RAM to disk temporarily (and then fetch it again later when needed). This process is expensive: when the OS needs to map a page to a physical representation, this is called a “page fault.” When the OS then realizes that it cannot map a page to RAM and you need to instead write an existing page to disk to free up space for this new page, you call this a “hard page fault.”
我们先谈谈物理表征。一般来说，一个页面可以映射到不同的东西：比如内存。可能是你想要方便寻址的硬件，有自己的内存。可能是磁盘上的一个文件。对于通用分配来说，有两个因素是关键的：一个是你的内存，另一个是你的页面文件（Linux 上的交换空间）。理想情况下，你需要的所有数据都放在 RAM 里，但很可能你的系统有时需要提供比内存更多的字节，所以它会暂时把部分 RAM 写到磁盘上（然后需要时再取回）。这一过程成本高昂：当操作系统需要将页面映射到物理表示时，这被称为“页面错误”。当操作系统意识到无法将页面映射到内存，而你需要写入已有页面到磁盘以释放空间时，你称之为“硬页错误”。

“Running out of memory” then can be multiple things:
“内存不足”可能有多重原因：

1. You could run out of virtual address space.
   你可能会用尽虚拟地址空间。
2. You could run out of physical representations for the physical pages you want to allocate.
   你可能会用完了你想分配的实体页面的物理表示。

The first kind of failure is very unlikely on a 64bit system. While your OS generally will not allow you to use all 64 bits for virtual addresses, you are probably looking at 48 bits or so, which is still multiple terabytes. The second failure case is much more insidious, primarily because it could happen *after* the OS has already handed you some virtual address space: It’s a feature of the system that you can get more virtual memory than you have physical bytes available.
第一种故障在 64 位系统上非常罕见。虽然你的操作系统通常不允许你用全部 64 位作为虚拟地址，但你大概率会看到 48 位左右，这仍然是数 TB。第二种失败情况更为隐蔽，主要因为它可能*发生在*操作系统已经给你一些虚拟地址空间之后：系统的一个特性是你能获得比物理字节更多的虚拟内存。

On Windows, this tension is resolved by inserting a step between allocating virtual memory and using it. Virtual memory management is expressed through `VirtualAlloc` and `VirtualFree`. `malloc`, `new`, or whatever means of allocation you are using are most likely going to call `VirtualAlloc` somewhere in their implementation at some point. This is the flow:
在 Windows 上，这种矛盾通过在虚拟内存分配和使用之间插入一步来解决。虚拟内存管理通过 `VirtualAlloc` 和 `VirtualFree` 实现。`malloc`、`new`，或者你用的任何分配方式，很可能在某个时候会调用 `VirtualAlloc`。流程如下：

- When you run `uint8_t* p = (uint8_t*) VirtualAlloc(NULL, 4096, MEM_RESERVE, PAGE_READWRITE)`, you reserve 4096 bytes worth of virtual address space, but it’s really just the address space, nothing else.
  当你运行 `uint8_t* p = （uint8_t*） VirtualAlloc（NULL， 4096， MEM_RESERVE， PAGE_READWRITE）` 时，你会保留 4096 字节的虚拟地址空间，但实际上只是地址空间，没有其他。
- When you run `p = VirtualAlloc(p, 4096, MEM_COMMIT, PAGE_READWRITE)`, you are telling the operating system that you intend on actually using it. You are asking the OS to *commit* to giving this memory a physical representation when you actually touch it. Note that we are reassigning `p` here.
  当你运行 `p = VirtualAlloc（p， 4096， MEM_COMMIT， PAGE_READWRITE）` 时，你是在告诉操作系统你打算实际使用它。你是在请求操作系统*承诺*在你实际触摸时赋予这段记忆一个物理表现。注意这里我们重新分配`了 p`。
- When you run `p[0] = 255`, you are actually touching the memory and it is likely only at this point that the operating will ensure that the page has a physical representation.
  当你运行 `p[0] = 255` 时，实际上是在触碰内存，只有在这时操作才会确保页面具有物理表示。

The most commonly used high-level allocation functions do not know the distinction, they always give you committed memory. Windows will happily let you allocate tons of virtual memory, but once you ask it to commit, it can fail: `VirtualAlloc` returns `NULL` if the amount of committed memory would exceed the combined size of your RAM and your page file. This means that one of the prime reasons for why your favorite allocation functions might return `NULL` on Windows is that the OS won’t commit to giving you more memory because the system-wide total size of committed memory exceeds the combined size of RAM and page file. Windows pushes out-of-memory conditions into your program through allocation failures[1](https://blog.s-schoener.com/2024-02-01-ooms/#fn:pagefilesetup).
最常用的高层分配函数并不区分，它们总是给你已承诺的内存。Windows 会很乐意让你分配大量虚拟内存，但一旦你要求提交，它可能会失败：如果提交的内存量超过你的内存和分页文件的总和，`VirtualAlloc` 会返回 `NULL`。这意味着你喜欢的分配函数在 Windows 上返回 `NULL` 的主要原因之一，是操作系统不会承诺给你更多内存，因为系统范围内已提交的内存总容量超过了 RAM 和分页文件的总和。Windows 通过分配失败将内存不足的条件推入你的程序 [1](https://blog.s-schoener.com/2024-02-01-ooms/#fn:pagefilesetup) 。

## So how much memory is left? 那么还剩多少内存？

Now, what APIs exist to see how much memory is still available? The functions you may to want to use are [GlobalMemoryStatusEx](https://learn.microsoft.com/en-us/windows/win32/api/sysinfoapi/nf-sysinfoapi-globalmemorystatusex), [GetPerformanceInfo](https://learn.microsoft.com/en-us/windows/desktop/api/psapi/nf-psapi-getperformanceinfo), and [GetProcessMemoryInfo](https://learn.microsoft.com/en-us/windows/win32/api/psapi/nf-psapi-getprocessmemoryinfo).
那么，有哪些 API 可以查看还有多少内存可用？你可能需要使用的函数有 [GlobalMemoryStatusEx](https://learn.microsoft.com/en-us/windows/win32/api/sysinfoapi/nf-sysinfoapi-globalmemorystatusex)、[GetPerformanceInfo](https://learn.microsoft.com/en-us/windows/desktop/api/psapi/nf-psapi-getperformanceinfo) 和 [GetProcessMemoryInfo](https://learn.microsoft.com/en-us/windows/win32/api/psapi/nf-psapi-getprocessmemoryinfo)。

On Windows, there are broadly five things that these APIs make measurable for you:
在 Windows 上，这些 API 大致可以为你衡量五件事：

1. You can measure currently available and total virtual memory (see `GlobalMemoryStatusEx`). That’s just “how much virtual address space did a process reserve?” On 64bit system you are in all likelihood not going to run out of virtual memory.
   你可以测量当前可用和总虚拟内存（参见 `GlobalMemoryStatusEx`）。这只是“进程保留了多少虚拟地址空间？”在 64 位系统上，你很可能不会用完虚拟内存。
2. You can measure the system-wide RAM usage, called “physical memory usage” (see `GlobalMemoryStatusEx`). This is not “memory that has a physical representation” but memory that is actually, physically in RAM. It does not include the pagefile. This value is naturally system-wide, because RAM is associated with the entire system. This is a useful value to track: not because you may fail to allocate, but because being close to your physical memory limit means that you are likely constantly running into hard pagefaults, which will grind programs to a halt.
   你可以测量系统范围的 RAM 使用情况，称为“物理内存使用率”（参见 `GlobalMemoryStatusEx`）。这不是“有物理表示的内存”，而是实际上物理上存在于 RAM 中的内存。它不包含分页文件。这个值自然是系统范围的，因为 RAM 与整个系统相关联。这是一个有用的数值：不是因为你可能没分配，而是因为接近物理内存极限意味着你很可能会不断遇到硬页面错误，导致程序停顿。
3. You can measure the system-wide committed memory and its limit (see `GlobalMemoryStatusEx` and `GetPerformanceInfo`). The naming for this is not consistent, some APIs call this “page file usage” or the “global commit charge.” This is a useful value to track, because every typical allocation you make is going to count against the commit value. If you only track one metric, track this. Note that `GetPerformanceInfo` reports the result in pages, not in bytes.
   你可以测量系统范围内已提交内存及其限制（参见 `GlobalMemoryStatusEx` 和 `GetPerformanceInfo`）。这种名称并不一致，有些 API 称之为“页面文件使用”或“全局提交费用”。这是一个有用的跟踪值，因为你所做的每一次典型分配都会计入提交值。如果你只追踪一个指标，就跟踪这个。注意，`GetPerformanceInfo` 以页面为单位报告结果，而非字节。
4. You can measure the process-specific “private bytes” (see `GetProcessMemoryInfo`). This is also called the “private usage” or the “process commit charge.” All regular allocations count towards this. This value is by itself can be an indicator for memory usage but it does not accurately represent how your process contributes to the thing that actually counts: the system-wide committed memory. First, your process could share memory with other processes. You can get an indication of the amount of shared memory as well via APIs, but that doesn’t necessarily help you in accounting since you still don’t know *which* processes share memory. Second, there are other (admittedly obscure) ways to increase committed memory, see [Pushing the Limits of Windows: Virtual Memory](https://techcommunity.microsoft.com/t5/windows-blog-archive/pushing-the-limits-of-windows-virtual-memory/ba-p/723750).
   你可以测量进程特定的“私有字节”（参见 `GetProcessMemoryInfo`）。这也被称为“私有使用”或“进程提交负荷”。所有常规分配都计入此项。这个数值本身可以作为内存使用量的指标，但它并不能准确反映你的进程如何贡献真正有意义的部分：系统范围的已提交内存。首先，你的进程可以与其他进程共享内存。你也可以通过 API 获得共享内存的数量，但这对会计并不一定有帮助，因为你仍然不知道*哪些*进程共享内存。其次，还有其他（虽然较为晦涩）的方法可以增加已提交内存，参见《[ 推动 Windows 的极限：虚拟内存 ](https://techcommunity.microsoft.com/t5/windows-blog-archive/pushing-the-limits-of-windows-virtual-memory/ba-p/723750)》。
5. You can measure the process-specific “working set size” (see `GetProcessMemoryInfo`). The working set is the set of pages of your process’ virtual address space that are currently mapped to RAM. The working set can include pages that are shared with other processes, so it is misleading to claim that this represents the amount of physical memory your process is responsible for.
   你可以测量进程特定的“工作集大小”（参见 `GetProcessMemoryInfo`）。工作集是当前映射到 RAM 的进程虚拟地址空间中的页面集合。工作集可以包含与其他进程共享的页面，因此声称这代表进程所占用的物理内存量是误导性的。

Note that committed memory does not generally correlate with physical memory or working set size. To illustrate this, consider these extreme cases:
注意，已提交的内存通常不与物理内存或工作集大小相关。举例来说，考虑以下极端情况：

- You can have very high committed memory usage but very low physical memory usage (and working set size), because you never touched the pages. You are still going to run out of memory.
  你可以有很高的承诺内存使用率，但物理内存占用（和工作集大小）却很低，因为你从未触碰过页面。你还是会用完内存。
- You can have a high physical memory usage but a very low committed memory usage. This is a bit counter-intuitive, but you can get this by manually memory mapping a file into memory: There’s no need to commit any memory. The file contents will still be loaded into RAM on access, but we can always write back to disk if we need space. There is no need for any additional physical backing (unless you use copy-on-write).
  你可以有很高的物理内存使用，但承诺的内存占用率却非常低。这有点反直觉，但你可以通过手动将文件映射到内存来实现：无需提交任何内存。访问时文件内容仍会加载到内存中，但如果需要空间，我们总能写回磁盘。不需要额外的物理备份（除非你使用写边复制）。

Great, so all you have to do is call the system APIs and be done with it, right? In general, yes. But when you run in a docker container, that is unfortunately insufficient. You can still call of these functions, but they do not give you the information you would hope to get. To elaborate a little bit, Windows docker containers can be run in two different isolation modes (see MSDN page [Isolation Modes](https://learn.microsoft.com/en-us/virtualization/windowscontainers/manage-containers/hyperv-container)): You can run in “process isolation mode” or in “HyperV isolation mode.” The latter essentially is just a VM. If you are running docker locally, you are likely using HyperV. If you run docker on a Windows server machine, you are likely running in “process isolation mode.” In both cases, the APIs above are going to produce wrong results (and you most might not even get to choose what isolation mode you need to use). In process isolation mode, you just get the stats of the host system[2](https://blog.s-schoener.com/2024-02-01-ooms/#fn:physicalmemory). In HyperV isolation mode, the APIs above will report the memory statistics of the virtual machine. That is going to be *close* to what you need, but not quite: The virtual machine always comes with a minimum amount of memory that is required to even run the system, and your container does not get to use the entirety of that memory. Both isolation modes use the same feature to enforce the memory limit: Windows job objects (see MSDN page [Job Objects](https://learn.microsoft.com/en-us/windows/win32/procthread/job-objects)).
太好了，所以你只需要调用系统 API，然后就完成了，对吧？总体来说，是的。但当你在 docker 容器中运行时，这显然是不够的。你仍然可以调用这些功能，但它们不会提供你期望获得的信息。稍微详细说明一下，Windows Docker 容器可以运行两种不同的隔离模式（参见 MSDN 页面[的隔离模式 ](https://learn.microsoft.com/en-us/virtualization/windowscontainers/manage-containers/hyperv-container)）：你可以选择“进程隔离模式”或“HyperV 隔离模式”。后者本质上就是虚拟机。如果你本地运行 docker，很可能是在用 HyperV。如果你在 Windows 服务器上运行 docker，很可能是“进程隔离模式”。在这两种情况下，上述 API 都会产生错误的结果（而且你甚至可能无法选择需要使用的隔离模式）。在进程隔离模式下，你只会获得主机系统的 [2](https://blog.s-schoener.com/2024-02-01-ooms/#fn:physicalmemory) 统计数据。在 HyperV 隔离模式下，上述 API 会报告虚拟机的内存统计数据。这会*接近*你需要的，但还不完全是：虚拟机总是自带运行系统的最低内存，而你的容器不能用完整个内存。两种隔离模式都使用相同的功能来强制内存限制：Windows 作业对象（参见 MSDN 页面 [“作业对象”](https://learn.microsoft.com/en-us/windows/win32/procthread/job-objects)）。

Job objects are basically nestable groups of processes. The two main APIs of interest here are [IsProcessInJob](https://learn.microsoft.com/en-us/windows/win32/api/jobapi/nf-jobapi-isprocessinjob) and [QueryInformationJobObject](https://learn.microsoft.com/en-us/windows/win32/api/jobapi2/nf-jobapi2-queryinformationjobobject). You can call both of these with a `NULL` job handle to represent the current process’s job, if it exists. Using `JobObjectExtendedLimitInformation` as an argument for `QueryInformationJobObject`, you can find the peak memory usage and the commit limit for your job. At this point, the rest of the story could be *really* simple: You use the `JobObjectMemoryUsageInformation = 28` argument to get the current memory usage… except that this is not exposed: There is seemingly no way to get the current memory usage for a job object[3](https://blog.s-schoener.com/2024-02-01-ooms/#fn:limits).
作业对象基本上是可嵌套的进程组。这里关注的两个主要 API 是 [IsProcessInJob](https://learn.microsoft.com/en-us/windows/win32/api/jobapi/nf-jobapi-isprocessinjob) 和 [QueryInformationJobObject](https://learn.microsoft.com/en-us/windows/win32/api/jobapi2/nf-jobapi2-queryinformationjobobject)。你可以用 `NULL` 作业句柄调用这两个进程，如果当前进程的作业存在的话。使用 `JobObjectExtendedLimitInformation` 作为 `QueryInformationJobObject` 的参数，你可以找到作业的峰值内存使用量和提交限制。此时，故事的其余部分其实可以*很*简单：你用 `JobObjectMemoryUsageInformation = 28` 参数来获取当前内存使用情况......但这并未被暴露：似乎没有办法获得作业对象 [3](https://blog.s-schoener.com/2024-02-01-ooms/#fn:limits) 当前的内存使用情况。

*EDIT*: This paragraph is new! There actually is a very simple way to track this value. My colleague Laura Hermanns noticed what I did not, namely that you can query `JobObjectLimitViolationInformation` even when there has not been any limit violation. So you simply pass that value to `QueryInformationJobObject` and read out the `JobMemory` field from the `JOBOBJECT_LIMIT_VIOLATION_INFORMATION` struct. Done!
*编辑* ：这段是新写的！实际上，有一个非常简单的方法可以追踪这个数值。我的同事 Laura Hermanns 注意到了我没注意到的，即即使没有任何限制违规，你也可以查询 `JobObjectLimitViolationInformation`。所以你只需把这个值传给 `QueryInformationJobObject`，然后从 `JOBOBJECT_LIMIT_VIOLATION_INFORMATION` 结构体读取 `JobMemory` 字段。完成！

But we know that this value is tracked: You can see that value in a kernel debugger. There is some internal Microsoft code [that uses this API in their HCS shim](https://github.com/microsoft/hcsshim/blob/a1319d51465c4503b30625df5bb45291d7a4600b/internal/winapi/jobobject.go#L58), which is how docker gets this information to be displayed in its stats. To do so you (seemingly) need to open the job object manually with the right permissions *and* run with elevated access – but so far I have failed to find a way to get an explicit handle to the job object from within the docker container, and running arbitrary code as administrator is not a good idea, even inside of a container (see this blog post [How to change the user account for Windows containers](https://techcommunity.microsoft.com/t5/itops-talk-blog/how-to-change-the-user-account-for-windows-containers/ba-p/3601571)). And that’s assuming you are willing to rely on undocumented Windows APIs. So, dear Satya, if you are reading this please tell someone to expose that API, thank you.

## Oh, and Linux 哦，还有 Linux

While we are talking about docker containers, there is some fun to be had when you take Windows code and move it to Linux (using Wine). Of course this is going to cause all sorts of problems, but there is one particular problem worth calling out. Linux handles out-of-memory conditions very differently, at least by default: Like Windows, Linux will also allow you to allocate more virtual memory than the system can actually physically provide. But unlike Windows, it will then also *commit* to more memory than it can physically provide. This strategy is called “overcommit.” Overcommit is great when you assume that most programs do not use most of the memory you ask for. Linux allows you to configure whether to use overcommit or not. There are three settings:
说到 docker 容器，当你把 Windows 代码迁移到 Linux（用 Wine）时，其实也很有趣。当然，这会引发各种问题，但有一个特别值得指出的问题。Linux 对内存不足的处理方式非常不同，至少默认是这样：和 Windows 一样，Linux 也允许你分配比系统实际能提供的更多的虚拟内存。但与 Windows 不同的是，它会*提交*比物理容量更多的内存。这种策略被称为“过度承诺”。当你假设大多数程序并未占用你要求的大部分内存时，过度提交是个好问题。Linux 允许你配置是否使用超提交。有三种设置：

1. Heuristic overcommit: Generally allow overcommit, but do some sanity checking on the incoming allocations (to catch cases that are most likely just wrong).
   启发式过度承诺：通常允许过度提交，但对新分配进行一些合理性检查（以发现很可能是错误的情况）。
2. Always overcommit. 总是过度投入。
3. Never overcommit. 永远不要过度投入。

Linux 2.6 added the “never overcommit” option and set the default to “heuristic overcommit.” See the [overcommit accounting documentation](https://www.kernel.org/doc/Documentation/vm/overcommit-accounting) for details.
Linux 2.6 增加了“永不过度提交”选项，并默认设置为“启发式过度提交”。详情请参见[超额承诺会计文档 ](https://www.kernel.org/doc/Documentation/vm/overcommit-accounting)。

Overcommit means that the location where you run out of memory changes drastically: Instead of getting a failure from your allocator (say, `NULL` from `malloc`), the OS now needs to handle the case that it is running out of memory which it has already committed to providing. For a program that checks the results of all allocations to detect out-of-memory conditions, overcommit is bad news: Instead of failing at an allocation site your program will now fail at a much later point when it touches the memory it allocated. This moves the symptom further away from the problem, if your program has been written with the opposite behavior in mind.
过度提交意味着内存耗尽的位置发生了剧烈变化：操作系统不再是从分配器获得失败（比如从 `malloc` 的 `NULL` 中获得），而是需要处理已经承诺提供内存不足的情况。对于一个通过检查所有分配结果来检测内存不足状况的程序来说，过度提交是坏消息：程序不再在分配点失败，而是在更晚的节点访问它分配的内存时失败。如果你的程序是以相反的行为为设计的，这会使症状远离问题。

Unsurprisingly, just switching the system setting to instead never overcommit causes a whole slew of new interesting failures because you are now using a setting that is not the default, which just means that it is less likely that that option received the same level of testing and exposure as the default. The problems are probably complex. Wine seems to generally still want to do the right thing. For example, `VirtualAlloc` with `MEM_RESERVE` is properly mapped to `mmap` with `PROT_NONE` (indicating that the memory you allocated may not be accessed) and correctly does not count as committed memory.
毫不意外，仅仅把系统设置切换成永不过度提交，就会导致一系列新的有趣失败，因为你现在使用的设置不是默认设置，这意味着该选项获得与默认设置同等测试和曝光的可能性降低。问题可能很复杂。葡萄酒似乎通常仍然想做正确的事。例如，带有 `MEM_RESERVE` 的 `VirtualAlloc` 正确映射到带有 `PROT_NONE` 的 `mmap`（表示你分配的内存可能无法被访问），正确地不计入已提交内存。

With overcommit enabled, OOM conditions now need to be handled by the OS, not the program. The Linux kernel’s “OOM killer” is then in charge of handling the out-of-memory conditions, and it will forcefully free up memory by heuristically selecting some process to kindly ask to exit, which it will then terminate if it does not comply. You can influence that heuristic and mark your programs as non-targets, but ultimately that is likely just getting some other vital process killed. If there are no valid targets, the kernel panics.
启用超载入后，OOM 条件现在需要由操作系统而非程序来处理。Linux 内核的“OOM 杀手”负责处理内存不足的状况，它会通过启发式地选择某个进程友好地请求退出，强制释放内存，如果不执行则终止该进程。你可以影响这个启发式方法，把你的程序标记为非目标，但归根结底，这很可能只是让其他重要进程被扼杀。如果没有有效目标，内核会慌乱。

I can see the upsides of this design, but one downside is that it again complicates the question of how much you can allocate: You can have all the memory, actually, if you don’t care about killing everyone else. Sarcasm aside, my adventures in this space have stopped short of actually having to answer how you would most effectively determine how much memory you can still safely allocate in a Linux container. I am cautiously optimistic that this is easier to solve than on Windows.
我能理解这种设计的优点，但一个缺点是它再次复杂化了你能分配多少内存的问题：实际上，如果你不在乎杀光其他人，你可以拥有所有的记忆。撇开讽刺不谈，我在这方面的冒险还没真正问到，如何最有效地确定 Linux 容器中还能安全分配多少内存。我谨慎乐观地认为这比 Windows 更容易解决。
