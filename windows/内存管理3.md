# Anti-debug with VirtualAlloc’s write watch 与 VirtualAlloc 写守望的反调试

A lesser-known feature of the Windows memory manager is that it can maintain write watches on allocations for debugging and profiling purposes. Passing the `MEM_WRITE_WATCH` flag to [VirtualAlloc](https://msdn.microsoft.com/en-us/library/windows/desktop/aa366887.aspx) “causes the system to track pages that are written to in the allocated region”. The [GetWriteWatch](https://msdn.microsoft.com/en-us/library/windows/desktop/aa366573.aspx) and [ResetWriteWatch](https://msdn.microsoft.com/en-us/library/windows/desktop/aa366874.aspx) APIs can be used to manage the watch counter. This can be (ab)used to catch out debuggers and hooks that modify memory outside the expected pattern.
Windows 内存管理器的一个较少为人知的特性是它可以对分配对象进行写监控，用于调试和分析。将 `MEM_WRITE_WATCH` 标志传递给 [VirtualAlloc](https://msdn.microsoft.com/en-us/library/windows/desktop/aa366887.aspx)“会使系统追踪在分配区域内写入的页面”。 可以使用 [GetWriteWatch](https://msdn.microsoft.com/en-us/library/windows/desktop/aa366573.aspx) 和 [ResetWriteWatch](https://msdn.microsoft.com/en-us/library/windows/desktop/aa366874.aspx) API 来管理 watch 计数器。这可以用来（滥用）发现那些超出预期模式修改内存的调试器和钩子。

There are four primary ways to exploit this feature.
利用这一特性主要有四种方式。

The first is a simple buffer manipulation check. Allocate a buffer with write watching enabled, write to it once, get the write count, and see if it’s greater than 1.
第一个是简单的缓冲区操作检查。分配一个启用写监控的缓冲区，写一次，获取写入次数，看看是否大于1。

The second is an API buffer manipulation check. Allocate a buffer with write watching enabled, pass it as a parameter to an API that expects a buffer, but pass invalid values to other parameters. If an API hook doesn’t check parameters properly, or manipulates parameters, it may write to the buffer. Check the number of writes to the buffer after the call, and if it’s nonzero then there’s a hook in place. Any API will do as long as it writes to some memory. A particularly good trick is to use an API where there’s some kind of count value passed as a reference – in the real API the value will likely not be set, thus producing no memory writes, but in a hook there’s a bigger likelihood that they’ll set some placeholder value regardless.
第二个是 API 缓冲区操作检查。分配一个启用写监控的缓冲区，将其作为参数传递给期望缓冲区的 API，但将无效值传递给其他参数。如果 API 钩子没有正确检查参数或操作参数，可能会写入缓冲区。检查调用后对缓冲区的写入次数，如果不是零，说明有钩子存在。只要写入内存，任何 API 都能做到。一个特别好的技巧是使用带有某种计数值作为引用传递的 API——在真实 API 中该值很可能不会被设置，因此不会产生内存写入，但在 hook 中，他们更可能设置某个占位值。

Third, we can use the buffer to store the result of some check we care about, e.g. [IsDebuggerPresent](https://msdn.microsoft.com/en-us/library/windows/desktop/ms680345.aspx). If the write count is one and the value in the buffer is `FALSE` then we can assume that there’s no debugger attached and nobody tampered with the result of the call (or skipped the call).
第三，我们可以用缓冲区存储我们关心的某个检查结果，例如 [IsDebuggerPresent](https://msdn.microsoft.com/en-us/library/windows/desktop/ms680345.aspx)。如果写入次数为 1，缓冲区值为 `FALSE`，我们可以假设没有连接调试器，也没有人篡改（或跳过调用）。

Finally, we can allocate some memory with RWX protection and write watching enabled, copy some anti-debug check there, call ResetWriteWatch to ensure the write counter is zeroed, execute our payload, then check the write count.
最后，我们可以分配一些带有 RWX 保护和写监控功能的内存，复制反调试检查，调用 ResetWriteWatch 确保写计数归零，执行负载，然后检查写入计数。

Obviously in all cases these checks themselves can be skipped over, but it’s not a well known trick and may be missed by novice reverse engineers.
显然，在所有情况下，这些检查本身都可以跳过，但这并不是一个广为人知的技巧，新手逆向工程师可能会忽略。
