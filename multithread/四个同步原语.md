#### 四个同步原语

------

# 

| 原语                    | 本质模型                  | 可复用 | 需要计数 | 是否可多阶段 | 是否会丢信号 | 典型用途             |
| ----------------------- | ------------------------- | ------ | -------- | ------------ | ------------ | -------------------- |
| `absl::Notification`    | 一次性事件（event）       | ❌      | ❌        | ❌            | ❌            | “开始了/结束了”通知  |
| `std::latch`            | 倒计数门闩（countdown）   | ❌      | ✅        | ❌            | ❌            | 等 N 个任务完成一次  |
| `std::barrier`          | 循环栅栏（phase barrier） | ✅      | ✅        | ✅            | ❌            | 多线程分阶段同步     |
| `std::binary_semaphore` | 令牌（token）             | ✅      | ⚠（0/1） | ❌            | ❌（有缓冲）  | 控制资源/限流/信号量 |

## 1️⃣ `absl::Notification` —— 一次性“事件开关”

```cpp
absl::Notification done;

done.Notify();              // 触发（一次）
done.WaitForNotification(); // 等待
```

👉 **语义：某件事“已经发生了”**

- 不需要 mutex
- 不会丢信号（先 Notify 再 Wait 也 OK）
- **只能触发一次**

📌 适合：

- 线程初始化完成
- shutdown 信号
- one-shot ready flag

------

## 2️⃣ `std::latch` —— 一次性“倒计数门”

```cpp
std::latch l(3);

worker() {
    // 做完
    l.count_down();
}

l.wait(); // 等 3 个完成
```

👉 **语义：等 N 个事件全部完成**

- 内部计数递减到 0 才放行
- 一次性（用完就废）

📌 适合：

- 等多个线程初始化完成
- 并行任务汇总

------

## 3️⃣ `std::barrier` —— 可复用“多阶段同步点”

```cpp
std::barrier b(3);

worker() {
    // phase 1
    b.arrive_and_wait();

    // phase 2
    b.arrive_and_wait();
}
```

👉 **语义：大家一起走到某个阶段，再一起继续**

- 每一轮称为一个 **phase**
- 所有线程都到达 → 一起放行
- **可以循环使用**

📌 适合：

- 并行算法（分阶段计算）
- SIMD/数值计算 pipeline

------

## 4️⃣ `std::binary_semaphore` —— “一个令牌”

```cpp
std::binary_semaphore sem(0);

sem.release(); // +1（最多 1）
sem.acquire(); // -1（没有就阻塞）
```

👉 **语义：是否有“许可”**

- 类似 mutex，但更通用
- 可以先 release，再 acquire（不会丢）
- 本质是 **0/1 计数器**

📌 适合：

- 控制资源访问
- 线程间信号（比 cv 简单）
- 替代部分 condition_variable 场景

------

# 🔍 关键差异（你真正要抓住的）

## 🔥 1. “事件 vs 计数 vs 阶段 vs 令牌”

| 类型             | 原语         |
| ---------------- | ------------ |
| 事件（发生一次） | Notification |
| 计数（等 N 个）  | latch        |
| 阶段（循环同步） | barrier      |
| 资源（许可）     | semaphore    |

------

## 🔥 2. 是否“一次性”

| 原语             | 是否可复用 |
| ---------------- | ---------- |
| Notification     | ❌          |
| latch            | ❌          |
| barrier          | ✅          |
| binary_semaphore | ✅          |

------

## 🔥 3. 是否需要“多个线程协同”

| 原语         | 是否多线程协作 |
| ------------ | -------------- |
| Notification | ❌（单事件）    |
| latch        | ✅              |
| barrier      | ✅（强同步）    |
| semaphore    | ⚠（弱同步）    |

------

## 🔥 4. 是否替代 condition_variable？

| 原语         | 能否替代 cv   |
| ------------ | ------------- |
| Notification | ✅（简单场景） |
| latch        | ❌             |
| barrier      | ❌             |
| semaphore    | ✅（很多场景） |

------

# 🎯 实战选型指南（直接用）

## ✔ 用 `absl::Notification`

当你只需要：

> 👉 “某个事件发生一次，别人等它”

------

## ✔ 用 `std::latch`

当你需要：

> 👉 “等 N 个线程完成一次”

------

## ✔ 用 `std::barrier`

当你需要：

> 👉 “多个线程分阶段同步（反复）”

------

## ✔ 用 `std::binary_semaphore`

当你需要：

> 👉 “控制访问 / 线程信号 / 替代 cv”

------

# 🚨 常见误用（很重要）

## ❌ 用 Notification 做队列同步

👉 错（只能一次）

------

## ❌ 用 latch 做循环任务

👉 错（不可复用）

------

## ❌ 用 barrier 做生产者-消费者

👉 错（语义不匹配）

------

## ❌ 用 condition_variable 替代 semaphore

👉 可以，但复杂度更高

------

# 🧠 一句话总结

> 👉

- `Notification` = **一次性事件**
- `latch` = **一次性计数器**
- `barrier` = **循环阶段同步**
- `binary_semaphore` = **资源令牌**
