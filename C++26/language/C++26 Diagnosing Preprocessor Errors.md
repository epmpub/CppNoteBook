诊断预处理器错误

keyword: 预处理器 相关

**C++26 中的“Diagnosing Preprocessor Errors”（诊断预处理器错误）** 是通过 **P2843R3**（“Preprocessing is never undefined”）提案引入的一项重要改进。

### 背景与动机
C++ 预处理器（preprocessor）阶段长期存在一些**未定义行为（Undefined Behavior, UB）**。这些 UB 发生在编译时（预处理阶段），理论上允许编译器做任何事情（如崩溃、生成错误代码等），但实际中很少被利用，且难以调试。

**P2843R3** 的核心目标是：

- 将预处理器中的 UB 替换为 **ill-formed, no diagnostic required (IFNDR)**（程序格式错误，但不强制要求诊断）。
- 进一步分析这些情况，**尽可能要求编译器发出诊断（错误或警告）**，让错误在预处理阶段就被明确捕获。
- 使预处理阶段“永不未定义”，提升编译时的安全性与可预测性。

这与之前 P2621（移除词法分析中的 UB）等提案一脉相承，进一步清理了翻译阶段的规范漏洞。

### 主要变化
提案针对多个常见/边缘的预处理器构造进行规范和诊断加强，包括：

- **`defined` 运算符的误用**（如宏替换后产生无效形式）。
- **宏参数中出现预处理指令**（之前 UB，现在明确 ill-formed 并倾向诊断）。
- **`#include` 指令宏展开后无效**。
- **`#`（stringize）和 `##`（token pasting/concatenation）产生无效 token 或字符串字面量。
- **预定义宏的重定义/undef**（如 `__LINE__`、`__FILE__` 等）。
- **`#line` 指令的无效形式**（如行号越界）。
- 其他如 ill-formed comments、保留标识符作为宏名等。

许多之前在 `-pedantic` 下才警告的情况，现在在 C++26 模式下**默认诊断**（或至少要求实现提供诊断）。

### 对开发者的影响
- **更好的错误诊断**：以前可能悄无声息地“工作”或导致神秘 bug 的代码，现在会在编译时明确报错，帮助及早发现问题。
- **兼容性**：现有代码如果依赖这些 UB，可能需要修复。编译器（如 GCC 16、Clang 等）已开始实现，并可能通过警告逐步过渡。
- **更安全的编译**：编译过程本身更可靠，不会因为预处理器 UB 而出现“鼻妖”（nasal demons）式行为。
- 与 `#error`、`#warning` 等现有诊断指令配合更好。

### 编译器支持
- **GCC 16**：支持（Diagnosing Preprocessor Errors）。
- 其他主流编译器（如 Clang、MSVC）也在跟进 C++26 特性。

更多细节可参考提案文档 P2843R3（可在 isocpp.org 或 open-std.org 找到）。这项特性是 C++26 在“安全性与诊断”方向的重要一步，与 Contracts、Erroneous Behavior 等特性共同提升语言的可靠性。
