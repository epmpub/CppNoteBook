**C++26 中的“Allowing attaching main to the global module”（允许将 main 附加到全局模块）** 是通过 **P3618R0** 提案引入的特性。

### 背景与动机
在 C++20 模块系统中，所有实体（entity）都必须“附加”（attached）到某个模块。**全局模块（global module）** 是特殊的隐式模块，用于存放未明确属于任何具名模块的代码（如传统的非模块代码、`#include` 产生的声明等）。

之前，**`main()` 函数** 有特殊限制：
- 不能直接在**具名模块（named module）** 的作用域中定义（会 ill-formed）。
- 也不能轻易使用 `extern "C++"` 等 linkage specification（之前对 `main` 是禁止的）。

这给某些场景带来不便，例如：
- **模块单元测试**：希望在模块实现单元中定义 `main()` 来测试该模块的非导出（non-exported）实体，而不污染全局命名空间或影响其他程序。

提案 P3618R0（作为 P3422 “允许 named module 中的 main” 的替代方案）通过**移除特殊限制**，让 `main` 能正常使用 `extern "C++"` 附加到全局模块，使语言规则更一致且不引入额外特例。

### 主要变化
- 允许在模块中这样写：

```cpp
module M;  // 或其他模块声明

extern "C++" int main() {  // 现在合法，main 附加到全局模块
    // 可以访问 M 中的非导出实体进行测试等
    test_internal_stuff();
}
```

- `main()` 仍保持实现定义的链接（linkage），但现在可以显式使用 `extern "C++"` 来控制其模块归属。
- 移除了之前对 `main` 的部分 linkage-specification 限制（只允许 `"C++"`）。
- 不改变 `main` 的其他语义（如不能是 `inline`、`static`、`constexpr`、`consteval`、协程、deleted 等）。

与之前相比，这是一种更“正统”的方式：利用已有的 `extern "C++"` 机制，而非为 `main` 添加新特殊规则。

### 对开发者的影响
- **更灵活的模块测试**：模块作者可以轻松编写测试用的 `main()`，无需额外技巧。
- **向后兼容**：现有代码不受影响；新代码更符合模块化理念。
- **一致性提升**：减少了语言中的特殊情况，`main` 的处理与其他函数更统一。
- **实现支持**：GCC 和 Clang 已部分支持（之前即使在旧标准下也常允许但带警告），C++26 模式下正式标准化。GCC 16 等版本已跟进。

这项特性是 C++26 模块系统完善的一部分，与之前的模块改进（如全局模块片段）协同工作，进一步提升模块的实用性。
