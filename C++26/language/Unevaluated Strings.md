## Unevaluated Strings

在 C++ 中，**未求值字符串（Unevaluated Strings）** 是 C++26 标准通过 **[P2361R6 提案](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2021/p2361r4.pdf)** 正式引入并规范化的一个核心概念。 [1, 2] 

简而言之，它们是写在代码里、但**不会在程序运行时占用内存，也不会被编译成二进制字符数组**的字符串字面量。它们仅仅作为**编译期**的元数据（Metadata）给编译器看。 [1, 3, 4] 

------

##  哪些属于“未求值字符串”？

在 C++ 中，很多地方的字符串并不是给运行时的用户看的，而是给**编译器**或链接器提供操作指令的： [1, 3] 

- **`static_assert` 的错误提示**：`static_assert(x > 0, "X must be positive");`
- **语言联动声明（Linkage）**：`extern "C" { ... }`
- **弃用/废弃属性（Attributes）**：`[[deprecated("Use new_func instead")]]`
- **字面量运算符名称**：`void operator ""_suffix(...);`
- **`#pragma` 运算符**：`_Pragma("pack(1)")` [1, 5, 6, 7, 8] 

------

## （C++26之前）的痛点与混乱

在 C++26 之前，标准没有对这些特殊上下文中的字符串做严格区分，导致了严重的逻辑混乱和编译器行为不一致：

## 1. 前缀的荒谬性

在旧标准中，下面这种写法在语法上居然是合法的：

```cpp
extern L"C" void foo(); // 使用了宽字符前缀 L
static_assert(false, u8"Error"); // 使用了 UTF-8 前缀 u8
```

这非常荒谬：`extern "C"` 只是告诉编译器使用 C 语言的链接规则，加一个宽字符前缀 `L` 或 `u8` 没有任何物理意义，但编译器却不得不去处理它的编码转换。 [5] 

## 2. 八进制/十六进制转义的不可控

如果写了 `static_assert(false, "\x123错误")`，这个 `\x123` 在不同的机器环境（甚至不同的目标运行字符集下）会被解析成完全不同的字节，导致编译期输出的错误提示变成一堆乱码。 [1, 5, 7] 

------

## 🚀 C++26 带来的改变与规范

为了解决上述混乱，C++26 正式在词法上将这些定义为 `unevaluated-string`，并强制执行以下三条铁律：

## 1. 严格禁止任何编码前缀（No Encoding Prefix）

未求值字符串**绝对不能**带有 `L`、`u8`、`u`、`U` 等前缀。 [3, 5] 

```cpp
extern L"C" {}          // ❌ C++26 编译报错！
static_assert(false, "OK"); // ✅ 只能使用最普通的无前缀字符串
```

## 2. 禁止数值转义（No Numeric Escape Sequences）

未求值字符串**绝对不能**包含八进制（如 `\123`）或十六进制（如 `\x4F`）转义字符。 [3, 7] 

- **原因**：这些转义深度依赖运行时的字符编码集（如 ASCII vs EBCDIC）。既然它是在编译期打印的，就不应该带有运行时编码。
- **例外**：允许使用通用字符名（如 `\u0041`）和简单转义（如 `\n`, `\t`, `\"`）。 [3, 7, 9] 

## 3. 它们不是表达式，不占内存

未求值字符串在语法上**不具有左值/右值等值类别（Value Category）**，也不会在程序的 `.rodata`（只读数据区）中产生任何实际的字符数组（如 `const char[N]`）。 [1, 4] 

------

## 

C++26 引入未求值字符串（Unevaluated Strings），成功解决了**编译期元数据与运行时代码逻辑混淆**的历史包袱。它让静态断言、属性提示等工具更加纯粹，确保了无论在什么编码环境下，编译器的报错和提示信息都能做到跨平台统一与安全。 [1, 3]
