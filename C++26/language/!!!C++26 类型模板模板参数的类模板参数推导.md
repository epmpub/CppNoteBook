**C++26 中的“Class Template Argument Deduction (CTAD) for type template template parameters”（类型模板模板参数的类模板参数推导）** 是通过 **P3865R3** 提案引入的特性（作为 Defect Report 处理）。

### 背景与动机
C++17 引入了 CTAD，允许在类模板对象初始化时省略模板参数，由编译器从构造函数参数推导。

之前，对于**模板模板参数（template template parameter）**，尤其是**类型模板模板参数**（如 `template<template<class...> class X>`），CTAD 不被核心语言正式支持，导致一些不一致：
- 标准库（如 C++23 的 `std::ranges::to`）已在规范中假设支持此功能。
- 实现实际上接受，但缺乏统一规则（CWG 3003 和 LWG 4381 问题）。

P3865R3 修复了这一 wording 缺陷，使 CTAD 正式适用于类型模板模板参数，并与 alias template 的 CTAD 语义保持一致。

### 主要变化
现在，在模板模板参数作为“占位符”时，可以进行 CTAD：

```cpp
template<typename T> struct C { C(T); };

template<template<typename> class X> void f() {
    X x(1);  // C++26 前可能 ill-formed 或依赖实现；现在 OK，推导为 C<int>
}

template void f<C>();  // 显式指定
```

**更复杂的例子**（考虑 variadic、默认参数等）：

```cpp
template<typename... Ts> struct Y {
    Y();
    Y(Ts...);
};

template<template<typename T = char> class X> void f() {
    X x0{};      // 推导 Y<char>（尊重模板模板参数的默认值）
    X x1{1};     // 推导 Y<int>
    X x2{1, 2};  // 错误：无法从 Y<int, int> 推导 X<T>（deducible check 失败，类似 alias template）
}

template void f<Y>();
```

- **处理机制**：将类型模板模板参数视为一个**假想的 alias template**，其 `defining-type-id` 指向实际的模板模板实参，然后应用现有的 CTAD 规则（包括隐式 deduction guides 和 deducible 检查）。
- 尊重模板模板参数上的**默认模板参数**。
- 处理**更特化（more specialized）** 的匹配规则（参考 P0522）。
- 非 deducible 的情况（如某些转换或不匹配）会被明确拒绝，保持一致性。

### 对开发者的影响
- **更好的模板元编程**：模板库作者可以更自然地在模板模板参数上使用 CTAD，尤其是在泛型容器/适配器中。
- **标准库一致性**：修复了 `std::ranges::to` 等功能的规范-实现差距。
- **向后兼容**：现有依赖实现的代码继续工作；新代码获得明确保证。
- **诊断改进**：不正确的 deduction 场景现在更清晰地 ill-formed。
- 这是一个 DR（Defect Report），意味着它可能 retroactively 应用于较早的标准模式。

### 编译器支持
- **Clang**：C++26 模式下正在跟进（P3865R3 标记为待实现）。
- GCC 和 MSVC 等也在逐步支持 C++26 相关改进。
